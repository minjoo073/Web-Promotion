## 0. 요구사항 재확인 (오해 방지)

강디가 설계 중인 핵심 인터랙션 5개의 기술 타당성을 자문한다:
- ① 복원 진행률 바
- ② "깨진→복원" 상태 전환
- ③ 후기웹 dashboard (무한스크롤·태그·리블로그)
- ④ 의도적 404 / glitch 연출
- ⑤ TikTok 챌린지(1차 유입) ↔ 사이트 연결

**가장 큰 비용 분기점은 ①·③·⑤가 공유하는 "복원률"이 *개인 진행*이냐 *전 팬 공유 누적*이냐다.** 이 한 가지 결정이 정적 사이트(서버 0)냐 실시간 백엔드(서버·DB·동기화)냐를 가른다 — 5장에서 두 시나리오로 갈라 비용을 제시한다.

> 공수 단위: **MD = man-day(1인 작업일)**. 풀스택 1명 기준 대략치. 비주얼 디자인·콘텐츠 제작 공수는 *제외*(개발만). 실제 견적은 콘텐츠 분량 확정 후 ±50% 변동 가능 [확인필요].

## 1. 결론 먼저 — 인터랙션 5개 구현 가능성 표

| # | 인터랙션 | 가능성 | 난이도 | 개발 공수(개발만) | 핵심 변수 |
|---|---|---|---|---|---|
| ① | 복원 진행률 바 | **가능** | 낮음(개인) / 높음(공유) | 0.5 MD(개인) / 5~8 MD(공유) | 개인 vs 공유 누적 (→ §3) |
| ② | 깨진→복원 상태 전환 | **가능** | 낮음~중 | 3~6 MD | 단순 CSS 전환이냐, 깨짐 자체를 "연출"로 정교화하냐 |
| ③ | 후기웹 dashboard(무한스크롤·태그·리블로그) | **조건부** | 중~높음 | 8~15 MD | 콘텐츠가 *고정 데이터*냐 *팬 UGC 실시간 반영*이냐 |
| ④ | 의도적 404 / glitch 연출 | **가능** (가드 필수) | 중 | 2~4 MD | SEO·공유 미리보기·봇 가드 안 하면 자해 (→ §4) |
| ⑤ | TikTok 챌린지 ↔ 사이트 연결 | **조건부** | 높음(자동) / 낮음(수동) | 1 MD(수동 집계) / 10~20 MD(자동 연동) | TikTok API로 자동 합산은 사실상 불가, 대안 필요 (→ §2.5) |

**한 줄 요약:** ①②④는 **순수 프론트엔드로 확실히 가능**. ③⑤와 "공유 복원률"이 백엔드를 부르는 지점이고, 여기가 전체 견적·일정·리스크의 90%다.

## 2. 인터랙션별 기술 자문

### ① 복원 진행률 바 — 가능
- **개인 시나리오:** 브라우저 `localStorage`에 진행값 저장, CSS `width`/`clip-path` 전환. 서버 0. **0.5 MD.** 새로고침해도 유지, 기기 바꾸면 리셋(로그인 없으면 당연).
- **공유 시나리오:** 전 팬 누적값을 서버에서 받아와 표시. 폴링 또는 실시간. 비용은 §3.
- 권장: 바 자체는 *표현*일 뿐이라 시각적 자유도 높음(캠코더 로딩바·복원 게이지 등 CEO 디자인 그대로 입힘). 숫자만 어디서 오느냐가 문제.

### ② "깨진→복원" 상태 전환 — 가능
- 기술적으로 두 레이어를 겹쳐두고(깨진 버전 / 복원 버전) opacity·mask·filter로 크로스페이드하는 게 가장 단순하고 안정적. **3~6 MD.**
- "깨짐" 연출은 두 갈래:
  - **싸고 안전:** 사전 제작된 *깨진 이미지/GIF 에셋*(디자이너가 미리 망가뜨린 JPEG·정지 GIF)을 복원 버전으로 교체. 런타임 부담 0. 모바일·저사양 안전. **권장.**
  - **비싸고 위험:** 실시간 셰이더/캔버스로 글리치를 *동적 생성*(WebGL). 멋지지만 저사양 모바일 발열·배터리·프레임드랍. MVP에서 비권장.
- 접근성: 깜빡임·플리커는 발작 유발 위험(WCAG 2.3.1, 초당 3회 번쩍임 금지). `prefers-reduced-motion`으로 정적 버전 폴백 필수. [Source: web.dev/learn/accessibility/motion]

### ③ 후기웹 dashboard (무한스크롤·태그·리블로그) — 조건부
- **무한스크롤 자체는 표준.** GSAP + Lenis 조합이 스크롤리텔링/무한스크롤의 사실상 업계 표준. [Source: tympanus.net/codrops 2026-05, Codrops "Infinite Scroll with GSAP & Lenis"]
- **조건이 갈리는 지점 = 콘텐츠 출처:**
  - **고정 데이터(권장 MVP):** 운영팀이 미리 넣은 포스트를 보여주는 "쇼" 형태의 dashboard. 리블로그/태그는 *연출된 인터랙션*(클릭하면 내 화면에 쌓이고 카운트만 올라감). 정적 배포 가능. **8~12 MD.**
  - **실시간 UGC(풀):** 팬이 실제로 올린 리블로그/밈을 공식 dashboard에 *반영*. → 업로드·검열(악성·저작권)·모더레이션·DB·스팸 방어 전부 필요. **+15~30 MD 및 상시 운영 인력.** 법적·운영 리스크 큼. docs/05 §3 "공동 큐레이션"은 매력적이지만 *풀 단계로 미뤄야* 한다.
- **성능 가드(저사양 모바일 필수):** 무한스크롤은 DOM이 무한정 쌓여 저사양폰에서 렉·메모리 폭증. **가상화(virtualization, 화면 밖 노드 언마운트)** 적용. [Source: uxpatterns.dev/patterns/navigation/infinite-scroll]
- **접근성 가드:** 무한스크롤은 스크린리더가 새 콘텐츠 로드를 인지 못하고 키보드 사용자가 위치를 잃음. "더 보기" 버튼 폴백 + ARIA live region 권장. [Source: uxpatterns.dev]

### ④ 의도적 404 / glitch 연출 — 가능하나 가드 필수
- 연출 자체는 쉬움. **위험은 "진짜로 깨진 것처럼 보이는 게 시스템·봇·공유에 진짜 피해를 준다"는 점.** (→ §4 상세)
- 핵심 원칙: **눈에는 404, HTTP 상태코드는 200.** "삭제됨" 화면을 보여주되 서버는 정상 응답해야 SEO·공유 미리보기가 안 깨진다. **2~4 MD** (가드 포함).

### ⑤ TikTok 챌린지 ↔ 사이트 연결 — 조건부 (자동 연동은 사실상 불가)
- 전략(docs/05)은 "챌린지 영상 게시 → 사이트 복원률에 합산". **TikTok이 제3자에게 '이 해시태그 영상 수'를 실시간·무료로 주는 공식 API는 없다.** Research API는 학술용·승인제·상업 캠페인 부적합. [확인필요 — TikTok for Developers 정책 직접 확인 권장]
- **현실적 대안 3종(쉬운 순):**
  1. **수동 집계(권장 MVP):** 운영팀이 해시태그 조회수/영상수를 주기적으로 확인해 복원률에 수동 반영. 서버 거의 0. **1 MD.** "실시간"은 아니지만 캠페인엔 충분.
  2. **사이트 측 행동으로 대체:** 복원 연료를 *틱톡 집계*가 아니라 *사이트 내 행동*(공유 버튼 클릭·인증 업로드·"내 챌린지 링크 제출")으로 정의. 측정 가능·조작 방어 가능. **권장 설계 변경.**
  3. **자동 스크래핑/연동:** 비공식·ToS 위반 위험·깨지기 쉬움. **비권장.**
- 결론: ⑤는 "틱톡 숫자를 실시간으로 빨아온다"가 아니라 **"틱톡=유입구, 복원 연료는 사이트에서 측정"**으로 재정의하면 비용·리스크가 급감한다. 이건 강기/아뱅과 합의 필요한 *설계 결정*.

## 3. ★ 핵심 — "복원률" 개인 vs 공유 누적 비용 차이

전체 견적을 가르는 단 하나의 결정. 두 시나리오:

| 항목 | A. 개인 진행 (내 복원률) | B. 전 팬 공유 누적 (집단 복원률) |
|---|---|---|
| 본질 | 내 브라우저 안의 게임 | 전 세계 팬이 함께 채우는 *하나의* 게이지 |
| 저장 위치 | `localStorage` (브라우저) | 서버 DB / Redis 카운터 |
| 백엔드 | **불필요 (서버 0)** | **필수** (DB + 실시간 동기화) |
| 배포 형태 | 순수 정적 사이트 (Vercel/Netlify 무료~저가) | 정적 프론트 + 백엔드 + 실시간 인프라 |
| 동시접속 부담 | 없음 (각자 따로) | **있음** — 발매일 트래픽 폭주가 직격 |
| 조작·치팅 방어 | 불필요 | **필수** (한 명이 매크로로 복원률 조작 방지) |
| 개발 공수(복원 로직만) | **0.5~1 MD** | **5~8 MD** (+ ③풀 UGC면 +15~30) |
| 월 운영비 | 거의 0 | 트래픽 따라 변동, 발매 스파이크 대비 필요 |
| 서사 강도 | 약함 ("나만의 진행") | **강함** ("우리가 함께 100% 복원" = docs/05 핵심) |

**기술적 권고 — "가짜 공유"라는 중간 지대:**
docs/05의 서사("전 팬덤 공동 달성")는 명백히 **B(공유 누적)**를 요구한다. 하지만 B의 풀버전은 비싸고 발매일 트래픽·치팅 방어가 부담이다. 그래서 **현실적 3단계 사다리**:

1. **B-lite (강력 권장 MVP):** 공유 카운터를 **단일 글로벌 숫자 하나만** 서버에 둔다. "복원 행동 1회 = 카운터 INCR(원자적 증가) +1", 화면엔 그 숫자를 폴링(수 초~수십 초마다 갱신)으로 표시. 개인별 기록·로그인 불필요. **Upstash Redis(서버리스) + 엣지 함수**면 서버 관리 없이 INCR 하나로 끝. [Source: upstash.com; systemdesign.one/distributed-counter-system-design — Redis INCR 원자적 카운터]. **공수 5 MD 안팎.** 진짜 "함께 채운다"는 서사를 *저비용으로* 산다.
2. **B-real-time (선택):** 폴링 대신 WebSocket으로 실시간 1초 단위 갱신. 숫자가 *살아 움직이는* 맛. 비용↑(상시 연결 인프라). [Source: ably.com — WebSocket+Redis Pub/Sub 스케일링]. 발매일 순간만 켜는 것도 방법.
3. **B-full (풀 단계로 연기):** 개인 로그인 + 기여 랭킹 + UGC 반영. 운영 인력 상주 필요.

> **결정 권고:** MVP는 **B-lite(글로벌 카운터 1개 + 폴링)**. 서사("우리가 함께")는 살리고, 발매일 트래픽·치팅·운영 부담은 풀 단계로 미룬다. 개인 진행 바(①)는 B-lite 위에 `localStorage`로 얹으면 "내 기여 + 전체 진행" 둘 다 공짜에 가깝게 보여줄 수 있다.

**치팅 방어(B 선택 시 필수):** 글로벌 카운터는 한 명이 스크립트로 무한 증가시킬 수 있다. 최소 가드 — IP/디바이스당 레이트리밋, 행동당 서버 검증, 비정상 급증 모니터링. 완벽한 방어는 비싸므로 "재미용 게이지"로 포지셔닝하고 *결정적 보상(미공개 트랙 해금)*은 운영팀 수동 확인 게이트를 두는 게 안전.

## 4. 성능 · 접근성 · 의도적-에러 리스크

### 4.1 의도적 404/glitch의 SEO·공유·봇 리스크 (가장 과소평가된 위험)
- **소셜 크롤러는 JavaScript를 실행하지 않는다.** 카카오톡·트위터·인스타 미리보기는 *최초 HTML*만 본다. 사이트가 "깨진 척"하려고 빈 HTML/에러를 주면 **공유 시 미리보기가 통째로 깨져 유입이 죽는다** (캠페인 자해). [Source: getrediate.com/blog/fix-broken-link-previews]
- **HTTP 상태코드와 화면을 분리하라:** 화면은 "this blog has been deleted"여도 **서버는 200 OK**를 반환하고, **Open Graph 태그(제목·썸네일·설명)는 정상 HTML에 박아둔다.** 그래야 공유 미리보기는 멀쩡하고 사람 눈에만 "깨진 폴더"로 보인다.
- **진짜 404가 필요한 트리거 페이지**(docs: 의도적 404 발굴)는 따로: 그 URL만 `noindex` + 캠페인 종료 후 정리 계획. 무분별한 404 남발은 "사이트 관리 부실" 신호로 검색 신뢰도를 깎는다. [Source: semrush.com/blog/site-crawler-errors; urllo.com 404 SEO guide]
- "영구 삭제" 서사를 *검색엔진에* 진짜로 전하고 싶을 때만 `410 Gone`. 마케팅 랜딩엔 쓰지 말 것. [Source: searchenginezine.com 404 vs 410]
- **봇/스캠 가드:** "깨진 링크처럼 보이는 정상 페이지"는 보안 스캐너·안티바이러스가 피싱으로 오탐할 수 있음. 공식 도메인·HTTPS·정상 메타 유지로 완화. [확인필요 — 카카오/네이버 공유 안전필터 동작은 실측 권장]

### 4.2 모바일·저사양 성능
- 주 유입이 **TikTok = 모바일·저사양 다수.** 무한스크롤(가상화 必)·glitch(사전 에셋 권장, WebGL 자제)·실시간(폴링 우선)을 §2 가이드대로 가면 부담 관리 가능.
- 발열·배터리: 상시 셰이더/캔버스 애니메이션은 5분만 켜둬도 저가폰이 뜨거워짐. 뷰포트 밖 애니메이션 정지(`IntersectionObserver`), 탭 비활성 시 정지 필수.

### 4.3 접근성 (WCAG)
- **`prefers-reduced-motion`**: glitch·플리커·무한스크롤 자동재생은 이 설정 켠 사용자에게 정적 폴백 제공. 저사양·저배터리 사용자에게도 이득. [Source: w3.org/WAI WCAG21 C39; developer.mozilla.org prefers-reduced-motion]
- **플래시 금지:** 초당 3회 이상 번쩍임은 발작 유발(WCAG 2.3.1). 캠코더 "적목 플래시" 연출은 *1회성·완만*하게.
- 무한스크롤 스크린리더/키보드 가드는 §2③ 참조.

## 5. 권장 스택 · 아키텍처

**한 줄:** **Next.js(정적/SSR 하이브리드) + GSAP ScrollTrigger + Lenis(스크롤) — 프론트 정적 배포(Vercel/Netlify), 공유 복원률만 Upstash Redis 글로벌 카운터(서버리스) 1개로 붙인다.**

- **왜 이게 가장 단순하면서 충분한가:**
  - 인터랙티브 내러티브·스크롤리텔링 사이트의 *실제 업계 표준 조합*이 Next.js + GSAP ScrollTrigger (+ Lenis 부드러운 스크롤). [Source: gsap.com/scroll; metabole.studio scrollytelling guide; Codrops 2026]
  - CEO가 **비주얼 디자인을 자유롭게 입힐 수 있다** — GSAP는 표현 레이어라 디자인 자유도 보존.
  - 프론트는 **정적 배포**(서버 0)라 발매일 트래픽에 강하고 싸다. 백엔드는 "복원률 카운터 1개"로 최소화 → Upstash 같은 서버리스로 *서버 관리 없이* INCR/GET만. 인프라 표면을 극단적으로 줄인다.
  - 실시간이 꼭 필요해지면(B-real-time) 그때 WebSocket(Ably/Upstash) 추가 — *나중에 얹는* 구조라 초기 결정을 묶지 않는다.
- **대안 검토(왜 안 골랐나):**
  - 풀 SPA(서버 상태 무거움) → 발매 트래픽·SEO·공유 미리보기에 불리.
  - WebGL/Three.js 전면 → 비주얼 천장은 높지만 저사양 모바일·공수·일정 부담. *부분적 글리치 연출에만 선택적으로*. [Source: medium scrollytelling libraries 2026]
  - 워드프레스/노코드 → 캠코더 글리치·무한스크롤·커스텀 복원 인터랙션을 원하는 결로 못 뽑음.

## 6. 롤백 / 단계화 (MVP → 풀)

| 단계 | 시점 | 포함 (개발) | 의도적 보류 |
|---|---|---|---|
