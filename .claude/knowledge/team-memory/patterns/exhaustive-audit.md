# 전수조사 (Exhaustive Audit) — 기능 감사·결함 발굴 방법론

> 출처: ClassCare(강사앱/학생앱) 기능 전수조사. 한 기능 단위에서 파일전송 26건+렉, 시험·퀴즈 15건처럼 *결함을 무더기로* 캐낸 작업 방식. 표면 점검("괜찮아 보임")이 아니라, 의심 경로를 코드로 끝까지 추적해 깨지는 지점을 증거와 함께 짚는다.

이 패턴은 **강개발(디버깅·감사)·강체크(회귀·발견)** 가 *기능 단위를 통째로 감사할 때* 쓴다. 단일 버그 리포트 대응이 아니라, "이 기능 전체를 점검하라"는 요청에 적용한다.

---

## 핵심 원칙 7

1. **전수(sweep), 표본 아님** — 그 단위의 *모든 경로·케이스*를 나열하고 하나씩 본다. 스팟체크로 "괜찮아 보임" 금지. (파일전송 26건은 표본이 아니라 전 경로를 훑어 나왔다.)
2. **증거 우선** — 모든 결함은 `파일:라인` + *재현 시나리오*로 제시. "위험할 수 있음" X / "`MainWindow.xaml.cs:19815` 브로드캐스트가 ACK 없이 fire-and-forget → 미수신 무알림" O.
3. **유형 일반화(bug-class sweep)** — 버그 1개를 찾으면, 같은 *유형*을 코드 전체에서 다시 훑는다. 한 증상 → 형제 전부. 자주 나오는 유형:
   - ACK 없는 fire-and-forget (전송했다 가정, 미수신 무알림)
   - 끊김/재접속 잔여물 (옛 세션 상태가 남아 새 세션 오염)
   - 틱/로컬시계 기반 시간 (`Environment.TickCount`·`DateTime.Now` 비교 → 기기·표준시 오차)
   - 세션/라우팅 꼬임 (중복 로그인·동명이인·옆반 섞임)
   - 빈 `catch{}` (에러 삼킴 → 침묵 실패)
   - 스냅샷 함정 (창 열 때 명단/상태를 복사해 두고, 그 뒤 변경이 반영 안 됨)
4. **양쪽 병렬 감사** — 프로토콜·기능에 두 주체가 있으면(강사앱↔학생앱, 송신↔수신, 웹↔앱) *양쪽을 병렬로* 본다. 한쪽 수정은 대개 반대쪽 미러 수정을 동반한다(예: 송신 측 ACK 추가 → 수신 측 ACK 응답).
5. **수정 전 진단 제시** — 코드를 건드리기 전에 *결함 목록 + 심각도*를 먼저 CEO/강팀장에게 보인다. 승인 후 일괄 수정. (CEO 상시 룰: "수정 시작할 땐 먼저 보여줘.")
6. **부수영향 분석 필수** — 모든 변경에 "이게 *기존에 잘 되던* 다른 기능을 깨나?"를 명시적으로 답한다. 신기능 추가 시 특히. (CEO 상시 우려: "새 기능이 기존 기능에 영향 안 가지?")
7. **계측으로 검증** — "빨라졌을 것"이 아니라 *실측*으로 닫는다(예: 렉 20~130ms 계측, 실기 13명 검증). 가정으로 완료 보고 금지.

## 절차 (기능 단위 1개)

1. **범위 한 줄** — 무슨 기능을, 강사앱·학생앱·웹 중 어디까지.
2. **양앱 병렬 감사** — 송·수신/양 주체를 동시에 훑어 결함 후보 수집(각 `파일:라인`).
3. **심각도 보고** — 🔴 치명(데이터 유실·집단 영향) / 🟡 주의(악용·간헐) / 🟢 경미. *수정 전에* 목록 제시.
4. **일괄 수정** — 승인된 결함을 유형별로 묶어 수정. 한쪽 고치면 반대쪽 미러도.
5. **빌드 + 검증** — 빌드 통과 + 가능하면 실측/실기 검증.
6. **문서** — 무엇을 왜 고쳤는지 핵심 결정만. 보류 항목은 *왜 보류인지* 한 줄.

---

## 금기

- **"괜찮아 보임"으로 종료** — 의심 경로를 코드로 안 따라갔으면 점검한 게 아니다.
- **증거 없는 결함 주장** — `파일:라인`·재현 없으면 보고하지 않는다.
- **한 버그만 고치고 끝** — 같은 유형을 전체에서 훑지 않으면 다음 세션에 또 터진다.
- **한쪽만 수정** — 프로토콜 반대쪽 미러를 빠뜨리면 비대칭 버그가 새로 생긴다.
- **진단 없이 바로 코드 수정** — CEO가 결함 목록을 먼저 보고 우선순위를 정한다.
- **부수영향 미분석 완료 보고** — "다른 기능 영향 없음"을 *확인*하기 전엔 끝이 아니다.
